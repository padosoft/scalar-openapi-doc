## 2026-06-13

- **[ci/review]** This repo has **two** automated PR reviewers configured: GitHub **Copilot** and a **Codex connector** (`chatgpt-codex-connector[bot]`). Codex auto-triggers on PR open / ready-for-review / `@codex review`. **Why:** the workflow contract said "Copilot review" but both bots comment, and their findings must both be resolved. **How to apply:** when waiting for reviews, poll for both bots; treat Codex findings as equally binding. You can nudge Codex with a `@codex review` PR comment.

- **[gh/copilot]** `gh pr edit <PR> --add-reviewer copilot` fails with `Could not resolve user with login 'copilot'`. **Why:** Copilot isn't a regular user login resolvable by `gh`. **How to apply:** use the GraphQL fallback (verified working on this repo):
  ```
  $prId = gh pr view <PR> --json id -q .id
  gh api graphql -f query='mutation RequestReviewsByLogin($pullRequestId: ID!, $botLogins: [String!], $union: Boolean!) { requestReviewsByLogin(input: {pullRequestId: $pullRequestId, botLogins: $botLogins, union: $union}) { clientMutationId } }' -F pullRequestId=$prId -F 'botLogins[]=copilot-pull-request-reviewer[bot]' -F union=true
  ```
  Then confirm with `gh api repos/padosoft/scalar-openapi-doc/pulls/<PR>/requested_reviewers` (Copilot should appear under `users`).

- **[git/windows]** Commits warn `LF will be replaced by CRLF` until a `.gitattributes` exists. **Why:** Git's autocrlf default on Windows. **How to apply:** `.gitattributes` with `* text=auto eol=lf` (+ binary rules) landed in PR #1; the warning is harmless once it's committed.

- **[docs/review]** A PR that *mandates reading files which don't exist yet* is internally inconsistent and both bots will flag it. **Why:** a fresh agent following the contract hits missing required files. **How to apply:** when a subtask defines rules that reference other deliverables, ship the referenced files in the same PR (or explicitly mark them forthcoming). Here, subtask 1.2's `docs/*` were folded into PR #1.

- **[copilot/autopilot]** `copilot --autopilot --yolo -p "/review …"` will *edit and commit fixes itself* unless told not to. Its first pass on the AGENTS.md diff auto-applied 6 fixes — most correct, but one rewrote a working GraphQL mutation into a broken `requestReviews` with empty `userIds`. **Why:** autopilot acts, doesn't just report. **How to apply:** for a report-only review, add "DO NOT modify or commit any files - report findings only" to the prompt; always re-verify any auto-applied change before trusting it. (Codified in AGENTS.md step 3.)

- **[git/review-base]** Review/diff a subtask branch against its **PR base** (the macro branch), not always `origin/main`. **Why:** subtask PRs target `task/<macro-name>`; diffing vs `origin/main` pulls in already-merged macro-branch changes as noise and makes the review gate ambiguous. Only the macro PR diffs vs `origin/main`. **How to apply:** `git diff origin/task/<macro-name>...HEAD` for subtasks. Flagged by both bots on PR #1.

- **[review/stale-threads]** Copilot/Codex **inline** comments anchor to a diff position from the commit they reviewed; after you add the referenced files in a later commit, the old threads persist verbatim ("docs/ doesn't exist") even though resolved. **Why:** GitHub doesn't auto-dismiss inline threads on follow-up commits. **How to apply:** don't chase stale inline threads — verify against the current tree, fix what's still real, and post one PR comment mapping stale-vs-fixed so the human reviewer isn't confused.

- **[review/false-positive-base]** Both bots flag files *added by the PR* as "does not exist in the repo yet" — repeatedly, every round. **Why (root cause):** they evaluate file existence against the **PR base / merge-base**, not the PR's resulting (merged) tree. When a subtask PR targets a fresh/empty macro branch, anything the PR introduces looks "missing" to them. **How to apply:** this is a categorical false positive — prove it with `git diff --name-status <base>...HEAD` (the files show as `A`dded), resolve the threads via GraphQL `resolveReviewThread`, and merge. Don't add "(forthcoming)" hedges to satisfy a base-vs-head confusion. When you can, put the dependency-defining files in the *same* PR as the rules that reference them (we folded docs/* into PR #1 for exactly this reason).

- **[git/pr-threads]** To clear bot threads programmatically: list them with the `reviewThreads` GraphQL query (gives `id`, `isResolved`, `isOutdated`, `path`), then `mutation { resolveReviewThread(input:{threadId:$t}){ thread{ isResolved } } }` per id. **Why:** GitHub's "auto-resolve on push" doesn't apply to bot inline threads; an unresolved thread blocks a clean merge state. **How to apply:** resolve outdated/false threads explicitly before `gh pr merge`.

- **[claude/powershell-is-a-separate-tool]** On Windows, Claude Code runs commands through the **PowerShell** tool, which is a **separate permission namespace** from `Bash`. `Bash(...)` allow/deny rules do **nothing** for PowerShell calls (and vice-versa). **Why:** a `Bash(git push*--force*)` deny gave a false sense of safety — the actual force-push via the PowerShell tool was unguarded, and the allowlist didn't reduce prompts. **How to apply:** mirror every rule under both tool names — `PowerShell(git status:*)`, `PowerShell(git push*--force*)`, etc. PowerShell rules use the same shape as Bash; aliases are canonicalized (`PowerShell(Get-ChildItem *)` also matches `gci`/`ls`/`dir`), matching is case-insensitive. `Read(...)`/`Edit(...)` deny rules cover the Read/Edit tools **and** recognized Bash file commands (`cat`/`head`/`tail`/`sed`), so `Read(./.env)` blocks both the Read tool and `Bash(cat .env)`. They do **not** cover arbitrary subprocess reads (a Python/Node script opening the file) and the docs don't list PowerShell `Get-Content` — but `Get-Content` isn't in the allowlist, so it prompts anyway. Source: https://code.claude.com/docs/en/permissions

- **[quality/strict-types-scaffold]** The Laravel React starter kit ships PHP files **without** `declare(strict_types=1)`, violating our "every PHP file" rule (Codex flagged it). Enforce it repo-wide with Pint instead of hand-editing: `pint.json` → `"rules": { "declare_strict_types": true }`, then `vendor/bin/pint` adds it to all files (56 here). **Why:** keeps the rule enforced by CI (`lint:check`) going forward. **How to apply:** after the sweep, re-run the **full test suite** — strict_types changes runtime scalar coercion and could surface `TypeError`s (here Pest stayed 39/39). Then **regenerate the PHPStan baseline** (`--generate-baseline`): Pint's reformatting plus any newly-surfaced trait-context return-type errors (e.g. `ProfileValidationRules::emailRules()`) need re-capturing for pristine starter code.

- **[ci/wayfinder-generated-files]** The React starter kit imports `@/routes`, `@/routes/*`, and `@/actions/*` which **Laravel Wayfinder generates** into `resources/js/{routes,actions,wayfinder}` — these are **gitignored**. So a clean CI checkout has no such modules: `tsc` fails with "Cannot find module '@/routes'" and ESLint `import/order` misclassifies them. **Why:** they only exist locally after a build/dev run; CI starts clean. **How to apply:** before `tsc`/`eslint` in CI, run `php artisan wayfinder:generate --with-form` (the `--with-form` flag is required — the Vite plugin uses `formVariants: true`, so without it the generated routes lack the `.form` property and `tsc` fails on `<route>.form` usage). This needs PHP + composer + `cp .env.example .env` + `key:generate` in the job. The `e2e` job gets them for free via `npm run build` (the Wayfinder Vite plugin runs during build).

- **[ci/vite-manifest-in-tests]** Pest **feature** tests render Inertia pages through `app.blade.php`, which calls `@vite` and throws `ViteManifestNotFoundException` (HTTP 500) when `public/build/manifest.json` is absent — which it is on a clean CI checkout that hasn't built assets. Locally it passes only because you previously ran `npm run build`. **Why:** false-green locally, red in CI. **How to apply:** call `$this->withoutVite()` in the base `TestCase::setUp()` so the suite never needs a built manifest. The `php` CI job then needs no Node/build step at all.

- **[ci/cross-platform-lockfile]** A `package-lock.json` generated on **Windows** is not `npm ci`-clean on **Linux CI**: npm records the host platform's optional deps (e.g. `@tailwindcss/oxide-win32`, `lightningcss-win32`, `rollup-win32`) but omits the Linux variants' transitive peers (`@emnapi/core`, `@emnapi/runtime`), so `npm ci` on ubuntu fails with "Missing: @emnapi/core from lock file". `npm ci --dry-run` **passes on Windows** (it only checks host-platform deps), so it gives false confidence. **Why:** Windows-dev + Linux-CI is our exact setup; Codex pushed us to `npm ci` but the Windows lock can't satisfy it on Linux. **How to apply:** use `npm install --no-audit --no-fund` in CI (tolerates platform diffs and reconciles the lock at install time) — this is what the Laravel starter did originally. The truly reproducible alternative is to generate the lockfile on Linux (WSL/CI) and commit that; revisit if reproducibility becomes critical.

- **[ci/duplicate-runs]** With both `on.push.branches` and `on.pull_request.branches` listing `task/**`, every push to a subtask branch that also has an open PR fires **two** identical workflow runs. **Why:** push event + pull_request event both match. **How to apply:** keep `push` for long-lived branches only (`main`/`master`/`develop`) and let `pull_request` cover `task/**`; add a `concurrency: { group: <workflow>-<ref>, cancel-in-progress: true }` block to cancel superseded runs.

- **[powershell/diff-encoding]** `git diff ... > file.txt` writes **UTF-16** under Windows PowerShell 5.1 (`>`/`Out-File` default), which the Copilot CLI review reads as garbage. pwsh 7+ defaults to UTF-8, but don't rely on the shell version. **Why:** a UTF-16 diff file silently breaks the local review gate for anyone on PS 5.1. **How to apply:** write UTF-8 explicitly — `git diff ... | Out-File -Encoding utf8 $env:TEMP\pr-diff.txt`. Codified in AGENTS.md step 3.

- **[claude/wildcard-anywhere]** Bash/PowerShell permission patterns allow `*` **at any position**, not only as a trailing `command:*`. `Bash(git push*--force*)` validly blocks `--force` anywhere in the command; `:*` is just shorthand for a trailing ` *`. **Why:** a reviewer claimed the deny "may never match without a colon" — false per the docs ("Wildcards can appear at any position"). **How to apply:** trust position-independent globs; the space-before-`*` rule still governs word boundaries (`ls *` ≠ `lsof`).

- **[claude/permission-precedence]** In `.claude/settings.json`, rules evaluate **deny → ask → allow**: a `deny` always wins, so you **cannot** allow-list an exception to a broad deny glob. A broad `Read(./.env.*)` deny blocks `.env.example`/`.env.testing` too, and re-adding them under `allow` does nothing. **Why:** an automated security review suggested "broad `.env.*` deny + narrow allow for safe files" — technically impossible here. **How to apply:** to protect secret env files while keeping only the **public template** (`.env.example`) readable, enumerate the secret-bearing patterns by Laravel convention (`.env`, `.env.local`, `.env.*.local`, `.env.testing`, `.env.<environment>`, `.env.backup/.bak`) rather than a catch-all glob. `.env.testing` is denied too — test env files can hold real test-DB creds and are typically gitignored; you *write* it (Edit/Write, not Read-denied), you don't need to read it back. (For how far `Read(...)` deny reaches across shell commands, see `[claude/powershell-is-a-separate-tool]` above — it covers the Read tool and recognized Bash file commands like `cat`/`head`, not arbitrary subprocess reads.)

- **[review/loop-termination]** A docs-only PR can loop forever if you treat every bot comment as binding — bots surface progressively more trivial nits and re-surface false positives. **Why:** the contract says "loop until zero comments," but "comments" must mean *actionable, true* findings. **How to apply:** terminate the loop when the local review is clean (NO FINDINGS) AND every remaining remote thread is provably false or already addressed (outdated). Document the rejection with proof, resolve, merge. Per receiving-code-review: verify findings, never implement false ones.

- **[schema/case-sensitive-strings]** On MySQL `utf8mb4_unicode_ci` (the project default), string UNIQUE indexes are **case-insensitive**: `/Pets` and `/pets` are treated as duplicates. OpenAPI path templates and tag names are case-sensitive identifiers. **Why:** correct grants for case-differing paths/tags would silently collide in production. **How to apply:** add `->collation('utf8mb4_bin')` to any column holding case-sensitive string identifiers (`tag`, `path`). Guard with `if (in_array(DB::getDriverName(), ['mysql', 'mariadb'], true))` — **include `mariadb`** (the framework reports it distinctly from `mysql`, and the project config defines both), so MariaDB also gets the binary collation; SQLite (binary UNIQUE by default) skips it to avoid an unknown-collation error.

- **[schema/audit-log-timestamps]** Use `->timestamp('created_at')->useCurrent()` (NOT NULL + DB default) for immutable audit tables, not `->nullable()`. **Why:** nullable `created_at` with no DB default lets any raw insert (bypassing Eloquent timestamping) create a row with no timestamp, weakening forensic ordering. **How to apply:** `useCurrent()` is SQLite-compatible and is the idiomatic Laravel pattern for append-only tables.

- **[schema/email-column-width]** When snapshotting a user's email into another table (e.g., auth_logs), use the **same width** as `users.email` (255), not 191. **Why:** 191 was the MySQL utf8mb4 index-length limit for the old MyISAM 767-byte row limit, but InnoDB 5.7+ / Laravel default uses 1000-byte, so 255 is safe. A 191-char snapshot can silently truncate or fail on a valid long email. **How to apply:** `$table->string('email', 255)` everywhere an email is stored, consistently with the `users` table.

- **[enum/http-verbs]** OpenAPI 3.x path-item objects support **8** HTTP methods: `get, put, post, delete, options, head, patch, trace`. Don't omit `TRACE`. **Why:** a TRACE operation in an upstream spec would be silently unrepresentable/ungrantable with a 7-verb enum. **How to apply:** `HttpVerb` must include `case Trace = 'TRACE'`; test `HttpVerb::values()` with `assertCount(8, ...)` and include `TRACE`.
