## 2026-06-13 (T4 hardening)

- **[openapi/server-url-defense-in-depth]** A service that injects values into the spec the browser renders must validate them itself, even when a FormRequest already validates upstream. Codex flagged that `injectServers` only checked non-empty string, so a seed/import/future caller bypassing the FormRequest could inject `javascript:alert(1)` or `not-a-url` into Scalar's playground server dropdown. **Why:** "the FormRequest validates it" is not a service-level guarantee ‚Äî the service promised "malformed entries are skipped." **How to apply:** validate server URLs at the injection point with `parse_url` requiring scheme+host and an allow-list of safe schemes (`http`/`https`); reject everything else with a logged warning. Defense-in-depth at the boundary that actually renders to the client.

- **[openapi/root-security-pruning]** When pruning `components.securitySchemes`, root-level `security` schemes are reachable **only if at least one surviving operation inherits root security** (i.e. has no own `security` key). Seeding them unconditionally keeps security metadata alive for a user filtered down to zero operations. **Why:** OpenAPI semantics ‚Äî an operation with its own `security` (including `security: []`, an explicit opt-out) overrides root; only operations *without* the key inherit root. A non-admin left with no operations inherits nothing. **How to apply:** in `securitySchemeNames`, collect operation-level schemes always, but seed root schemes only when some surviving op lacks the key (`inheritsRootSecurity()` helper). **Pair it with dropping the top-level `security` field** in `filterForUser` when nothing inherits it (Codex follow-up): otherwise pruning removes the scheme but the dangling root `security` requirement remains, referencing a scheme no longer in `components` ‚Üí an invalid spec. Test both: empty grants ‚Üí no `components` AND no `security`; a survivor that overrides ‚Üí root dropped, its own scheme kept; a survivor that inherits ‚Üí root `security` kept.

- **[openapi/host-allowlist-never-disabled]** An "empty allow-list = allow anything" fallback is a silent SSRF hole, even if the config *comment* claims "empty = only the upstream host is trusted" (the code didn't match the comment). Codex flagged that blank `OPENAPI_ALLOWED_HOSTS` (the `.env.example` default) skipped host validation entirely. **Why:** a defaults-on-blank that opens the gate contradicts a stated scheme+host rule and trusts every deployment to remember to populate the env var. **How to apply:** when the configured host list is empty, **derive it from the configured upstream URL's host** (never "any host"), and **fail closed** if no host can be resolved ‚Äî so host validation is always enforced and the code matches the documented contract. (Here it's also defense-in-depth: we only ever fetch the single `upstream_url` and disabled redirects, so the check is non-bypassing rather than a hard gate ‚Äî but it's robust if a future change varies the fetched URL.)

- **[security/pruning-is-not-optional]** A per-user spec-filtering proxy CANNOT offer a "keep orphan components" toggle (`prune_components=false`): retaining unreachable components either (a) leaks the definitions of ungranted operations/schemas, or (b) ‚Äî if you prune only the operation-bearing buckets (pathItems) ‚Äî leaves **dangling `$ref`s** from the buckets you kept (e.g. `components.callbacks` ‚Üí a pruned `pathItems`). Codex flagged both halves across successive rounds. **Why:** the two requirements (no-leak + no-dangling) are only simultaneously satisfiable by full reachability pruning. **How to apply:** make pruning a **security invariant, always applied** to the filtered (non-admin) spec; remove the disable toggle (config key, env var, and the "keep all" test). Admins bypass filtering entirely (`admin_sees_all`), so they still get the full spec ‚Äî the toggle had no safe use. Compute ONE full transitive closure (seeded from paths+webhooks+root/op security-scheme names, following every `$ref` incl. callbacks‚ÜípathItems) and keep only reachable entries.

- **[php/no-array-union-in-loop]** `docs/RULES.md` (ADR-07) forbids `$a += $b` / `array_merge` for building arrays inside a loop. When resolving a `$ref` chain that needs "closer link wins" precedence, don't `+=` per iteration: **collect each link as a layer** (`$layers[] = $link`, closest first) and merge once after the loop with `array_replace(...array_reverse($layers))` ‚Äî array_replace lets later args win, so reversing puts the highest-precedence (closest) layer last. Flagged by Codex on `resolvePathItem`.

- **[openapi/path-item-ref-reuse]** OpenAPI 3.1 lets a `paths`/`webhooks` entry be a Reference Object (`{"$ref": "#/components/pathItems/X"}`) reusing a shared path item. A naive per-operation filter that iterates a path item's fields sees only `$ref` (no HTTP-verb key), keeps it as a non-operation key, then drops the whole entry for having no surviving verb ‚Äî so granted reused operations silently vanish for non-admins (Codex). **Why:** the filter and the grant metadata (`extractTags`/`extractEndpoints`) both enumerate operations, and a `$ref`-only item yields none. **How to apply:** resolve path-item `$ref`s against `components.pathItems` BEFORE filtering/enumerating (a `resolvePathItem` helper), then **inline** the filtered survivors ‚Äî inlining (not keeping the `$ref`) is what prevents leaking the ungranted operations still present in the shared component. Resolve in all three enumerators (filter + extractTags + extractEndpoints) so referenced ops stay grantable, matching the [[openapi/webhook-path-collision]] metadata-vs-filter consistency rule. **Follow the WHOLE `$ref` chain (cycle-safe) and ensure NO path-item `$ref` survives in the output** (P1 follow-up): single-level resolution leaves a nested `$ref` in the filtered item, which `pruneComponents` then follows and keeps the entire (unfiltered) referenced path item ‚Äî re-leaking ungranted operations. Drop dangling/cyclic refs too. Test a chain `A ‚Üí ($ref B) + granted op`, B = ungranted op: assert B and the whole `components.pathItems` bucket are gone after filtering.

- **[review/recount-threads-after-each-round]** After a bot review round, **re-query the full unresolved-thread list** (not just the count from your monitor, and not just the first finding you read) before merging. A round can post multiple inline threads; here a `c75d983` round had TWO P2 threads (paths-required + path-item-$ref) but only one was actioned, and PR #14 was merged with the second still open. **Why:** monitors report a count, but you must read every thread body to disposition it. **How to apply:** immediately before `gh pr merge`, run the `reviewThreads(first:N){nodes{isResolved ‚Ä¶}}` query and assert ZERO unresolved (or that each open one is consciously declined-with-rationale). If a genuine finding slipped through post-merge, fix it in a follow-up subtask PR into the macro branch ‚Äî which is what `task/openapi-service-4-8-pathitem-refs` is.

- **[openapi/webhook-path-collision]** OpenAPI 3.1 `webhooks` and `paths` are separate maps, but path keys MUST start with `/` while webhook keys are arbitrary strings ‚Äî so a webhook can be keyed identically to a real path (both `/orders`). If endpoint grants use a bare `METHOD key`, a `POST /orders` path grant also matches the `/orders` webhook (Codex). **Why:** the two containers share an address space, so a grant for one silently exposes the other. **How to apply:** namespace webhook grant keys (`webhook:` prefix) in BOTH the matcher (`filterPathItemMap`, container-aware) and the metadata (`extractEndpoints` emits the namespaced `path` + a "(webhook)" label) so the address spaces are disjoint. Define the canonical grant-key format in the SERVICE (T4), before the data model/UI (T6) builds on it ‚Äî retrofitting the key format after grants are stored is far costlier. The webhook `path` stored in `user_allowed_endpoints` is just the namespaced string; no migration needed.

- **[openapi/3.1-paths-optional]** In **OpenAPI 3.1**, `paths` is **optional** (it was required in 3.0); a valid 3.1 document may have only `webhooks` or only `components`. So spec validation must require `openapi` + `info` + **at least one of** `paths`/`webhooks`/`components` ‚Äî NOT `paths` specifically. **Why:** a reviewer (Codex) flagged `assertValidSpec` for "not requiring paths," which is correct for 3.0 but would reject valid 3.1 webhook-only/components-only specs that this service deliberately supports. **How to apply:** keep the `$hasContainer` (paths OR webhooks OR components) check; decline any "require paths" suggestion with the 3.1 spec rationale. This is a recurring false-positive shape ‚Äî bots reason from 3.0 rules; cite the 3.1 OpenAPI Object fixed-fields (paths no longer in the required set) and resolve.

- **[phpstan/memory-limit-windows]** PHPStan at `level: max` on this app exhausts the default 128M PHP memory limit and crashes with `Child process error (exit code 255) while running parallel worker` (the parallel worker just hides the real OOM). **Why:** Herd's CLI php.ini caps `memory_limit` low; the crash message doesn't mention memory. **How to apply:** run `php -d memory_limit=1G vendor/bin/phpstan analyse`; if you see the opaque parallel-worker error, re-run single-threaded (`--debug`) to surface the true cause before assuming a code bug. (CI's `setup-php` doesn't hit this ‚Äî its default limit is higher.)

## 2026-06-13

- **[ci/review]** This repo has **two** automated PR reviewers configured: GitHub **Copilot** and a **Codex connector** (`chatgpt-codex-connector[bot]`). Codex auto-triggers on PR open / ready-for-review / `@codex review`. **Why:** the workflow contract said "Copilot review" but both bots comment, and their findings must both be resolved. **How to apply:** when waiting for reviews, poll for both bots; treat Codex findings as equally binding. You can nudge Codex with a `@codex review` PR comment.

- **[gh/copilot]** `gh pr edit <PR> --add-reviewer copilot` fails with `Could not resolve user with login 'copilot'`. **Why:** Copilot isn't a regular user login resolvable by `gh`. **How to apply:** use the GraphQL fallback (verified working on this repo):
  ```
  $prId = gh pr view <PR> --json id -q .id
  gh api graphql -f query='mutation RequestReviewsByLogin($pullRequestId: ID!, $botLogins: [String!], $union: Boolean!) { requestReviewsByLogin(input: {pullRequestId: $pullRequestId, botLogins: $botLogins, union: $union}) { clientMutationId } }' -F pullRequestId=$prId -F 'botLogins[]=copilot-pull-request-reviewer[bot]' -F union=true
  ```
  Then confirm with `gh api repos/padosoft/scalar-openapi-doc/pulls/<PR>/requested_reviewers` (Copilot should appear under `users`).

- **[git/windows]** Commits warn `LF will be replaced by CRLF` until a `.gitattributes` exists. **Why:** Git's autocrlf default on Windows. **How to apply:** `.gitattributes` with `* text=auto eol=lf` (+ binary rules) landed in PR #1; the warning is harmless once it's committed.

- **[docs/review]** A PR that *mandates reading files which don't exist yet* is internally inconsistent and both bots will flag it. **Why:** a fresh agent following the contract hits missing required files. **How to apply:** when a subtask defines rules that reference other deliverables, ship the referenced files in the same PR (or explicitly mark them forthcoming). Here, subtask 1.2's `docs/*` were folded into PR #1.

- **[copilot/autopilot]** `copilot --autopilot --yolo -p "/review ‚Ä¶"` will *edit and commit fixes itself* unless told not to. Its first pass on the AGENTS.md diff auto-applied 6 fixes ‚Äî most correct, but one rewrote a working GraphQL mutation into a broken `requestReviews` with empty `userIds`. **Why:** autopilot acts, doesn't just report. **How to apply:** for a report-only review, add "DO NOT modify or commit any files - report findings only" to the prompt; always re-verify any auto-applied change before trusting it. (Codified in AGENTS.md step 3.)

- **[git/review-base]** Review/diff a subtask branch against its **PR base** (the macro branch), not always `origin/main`. **Why:** subtask PRs target `task/<macro-name>`; diffing vs `origin/main` pulls in already-merged macro-branch changes as noise and makes the review gate ambiguous. Only the macro PR diffs vs `origin/main`. **How to apply:** `git diff origin/task/<macro-name>...HEAD` for subtasks. Flagged by both bots on PR #1.

- **[review/stale-threads]** Copilot/Codex **inline** comments anchor to a diff position from the commit they reviewed; after you add the referenced files in a later commit, the old threads persist verbatim ("docs/ doesn't exist") even though resolved. **Why:** GitHub doesn't auto-dismiss inline threads on follow-up commits. **How to apply:** don't chase stale inline threads ‚Äî verify against the current tree, fix what's still real, and post one PR comment mapping stale-vs-fixed so the human reviewer isn't confused.

- **[review/false-positive-base]** Both bots flag files *added by the PR* as "does not exist in the repo yet" ‚Äî repeatedly, every round. **Why (root cause):** they evaluate file existence against the **PR base / merge-base**, not the PR's resulting (merged) tree. When a subtask PR targets a fresh/empty macro branch, anything the PR introduces looks "missing" to them. **How to apply:** this is a categorical false positive ‚Äî prove it with `git diff --name-status <base>...HEAD` (the files show as `A`dded), resolve the threads via GraphQL `resolveReviewThread`, and merge. Don't add "(forthcoming)" hedges to satisfy a base-vs-head confusion. When you can, put the dependency-defining files in the *same* PR as the rules that reference them (we folded docs/* into PR #1 for exactly this reason).

- **[git/pr-threads]** To clear bot threads programmatically: list them with the `reviewThreads` GraphQL query (gives `id`, `isResolved`, `isOutdated`, `path`), then `mutation { resolveReviewThread(input:{threadId:$t}){ thread{ isResolved } } }` per id. **Why:** GitHub's "auto-resolve on push" doesn't apply to bot inline threads; an unresolved thread blocks a clean merge state. **How to apply:** resolve outdated/false threads explicitly before `gh pr merge`.

- **[claude/powershell-is-a-separate-tool]** On Windows, Claude Code runs commands through the **PowerShell** tool, which is a **separate permission namespace** from `Bash`. `Bash(...)` allow/deny rules do **nothing** for PowerShell calls (and vice-versa). **Why:** a `Bash(git push*--force*)` deny gave a false sense of safety ‚Äî the actual force-push via the PowerShell tool was unguarded, and the allowlist didn't reduce prompts. **How to apply:** mirror every rule under both tool names ‚Äî `PowerShell(git status:*)`, `PowerShell(git push*--force*)`, etc. PowerShell rules use the same shape as Bash; aliases are canonicalized (`PowerShell(Get-ChildItem *)` also matches `gci`/`ls`/`dir`), matching is case-insensitive. `Read(...)`/`Edit(...)` deny rules cover the Read/Edit tools **and** recognized Bash file commands (`cat`/`head`/`tail`/`sed`), so `Read(./.env)` blocks both the Read tool and `Bash(cat .env)`. They do **not** cover arbitrary subprocess reads (a Python/Node script opening the file) and the docs don't list PowerShell `Get-Content` ‚Äî but `Get-Content` isn't in the allowlist, so it prompts anyway. Source: https://code.claude.com/docs/en/permissions

- **[quality/strict-types-scaffold]** The Laravel React starter kit ships PHP files **without** `declare(strict_types=1)`, violating our "every PHP file" rule (Codex flagged it). Enforce it repo-wide with Pint instead of hand-editing: `pint.json` ‚Üí `"rules": { "declare_strict_types": true }`, then `vendor/bin/pint` adds it to all files (56 here). **Why:** keeps the rule enforced by CI (`lint:check`) going forward. **How to apply:** after the sweep, re-run the **full test suite** ‚Äî strict_types changes runtime scalar coercion and could surface `TypeError`s (here Pest stayed 39/39). Then **regenerate the PHPStan baseline** (`--generate-baseline`): Pint's reformatting plus any newly-surfaced trait-context return-type errors (e.g. `ProfileValidationRules::emailRules()`) need re-capturing for pristine starter code.

- **[ci/wayfinder-generated-files]** The React starter kit imports `@/routes`, `@/routes/*`, and `@/actions/*` which **Laravel Wayfinder generates** into `resources/js/{routes,actions,wayfinder}` ‚Äî these are **gitignored**. So a clean CI checkout has no such modules: `tsc` fails with "Cannot find module '@/routes'" and ESLint `import/order` misclassifies them. **Why:** they only exist locally after a build/dev run; CI starts clean. **How to apply:** before `tsc`/`eslint` in CI, run `php artisan wayfinder:generate --with-form` (the `--with-form` flag is required ‚Äî the Vite plugin uses `formVariants: true`, so without it the generated routes lack the `.form` property and `tsc` fails on `<route>.form` usage). This needs PHP + composer + `cp .env.example .env` + `key:generate` in the job. The `e2e` job gets them for free via `npm run build` (the Wayfinder Vite plugin runs during build).

- **[ci/vite-manifest-in-tests]** Pest **feature** tests render Inertia pages through `app.blade.php`, which calls `@vite` and throws `ViteManifestNotFoundException` (HTTP 500) when `public/build/manifest.json` is absent ‚Äî which it is on a clean CI checkout that hasn't built assets. Locally it passes only because you previously ran `npm run build`. **Why:** false-green locally, red in CI. **How to apply:** call `$this->withoutVite()` in the base `TestCase::setUp()` so the suite never needs a built manifest. The `php` CI job then needs no Node/build step at all.

- **[ci/cross-platform-lockfile]** A `package-lock.json` generated on **Windows** is not `npm ci`-clean on **Linux CI**: npm records the host platform's optional deps (e.g. `@tailwindcss/oxide-win32`, `lightningcss-win32`, `rollup-win32`) but omits the Linux variants' transitive peers (`@emnapi/core`, `@emnapi/runtime`), so `npm ci` on ubuntu fails with "Missing: @emnapi/core from lock file". `npm ci --dry-run` **passes on Windows** (it only checks host-platform deps), so it gives false confidence. **Why:** Windows-dev + Linux-CI is our exact setup; Codex pushed us to `npm ci` but the Windows lock can't satisfy it on Linux. **How to apply:** use `npm install --no-audit --no-fund` in CI (tolerates platform diffs and reconciles the lock at install time) ‚Äî this is what the Laravel starter did originally. The truly reproducible alternative is to generate the lockfile on Linux (WSL/CI) and commit that; revisit if reproducibility becomes critical.

- **[ci/duplicate-runs]** With both `on.push.branches` and `on.pull_request.branches` listing `task/**`, every push to a subtask branch that also has an open PR fires **two** identical workflow runs. **Why:** push event + pull_request event both match. **How to apply:** keep `push` for long-lived branches only (`main`/`master`/`develop`) and let `pull_request` cover `task/**`; add a `concurrency: { group: <workflow>-<ref>, cancel-in-progress: true }` block to cancel superseded runs.

- **[powershell/diff-encoding]** `git diff ... > file.txt` writes **UTF-16** under Windows PowerShell 5.1 (`>`/`Out-File` default), which the Copilot CLI review reads as garbage. pwsh 7+ defaults to UTF-8, but don't rely on the shell version. **Why:** a UTF-16 diff file silently breaks the local review gate for anyone on PS 5.1. **How to apply:** write UTF-8 explicitly ‚Äî `git diff ... | Out-File -Encoding utf8 $env:TEMP\pr-diff.txt`. Codified in AGENTS.md step 3.

- **[claude/wildcard-anywhere]** Bash/PowerShell permission patterns allow `*` **at any position**, not only as a trailing `command:*`. `Bash(git push*--force*)` validly blocks `--force` anywhere in the command; `:*` is just shorthand for a trailing ` *`. **Why:** a reviewer claimed the deny "may never match without a colon" ‚Äî false per the docs ("Wildcards can appear at any position"). **How to apply:** trust position-independent globs; the space-before-`*` rule still governs word boundaries (`ls *` ‚âÝ `lsof`).

- **[claude/permission-precedence]** In `.claude/settings.json`, rules evaluate **deny ‚Üí ask ‚Üí allow**: a `deny` always wins, so you **cannot** allow-list an exception to a broad deny glob. A broad `Read(./.env.*)` deny blocks `.env.example`/`.env.testing` too, and re-adding them under `allow` does nothing. **Why:** an automated security review suggested "broad `.env.*` deny + narrow allow for safe files" ‚Äî technically impossible here. **How to apply:** to protect secret env files while keeping only the **public template** (`.env.example`) readable, enumerate the secret-bearing patterns by Laravel convention (`.env`, `.env.local`, `.env.*.local`, `.env.testing`, `.env.<environment>`, `.env.backup/.bak`) rather than a catch-all glob. `.env.testing` is denied too ‚Äî test env files can hold real test-DB creds and are typically gitignored; you *write* it (Edit/Write, not Read-denied), you don't need to read it back. (For how far `Read(...)` deny reaches across shell commands, see `[claude/powershell-is-a-separate-tool]` above ‚Äî it covers the Read tool and recognized Bash file commands like `cat`/`head`, not arbitrary subprocess reads.)

- **[review/loop-termination]** A docs-only PR can loop forever if you treat every bot comment as binding ‚Äî bots surface progressively more trivial nits and re-surface false positives. **Why:** the contract says "loop until zero comments," but "comments" must mean *actionable, true* findings. **How to apply:** terminate the loop when the local review is clean (NO FINDINGS) AND every remaining remote thread is provably false or already addressed (outdated). Document the rejection with proof, resolve, merge. Per receiving-code-review: verify findings, never implement false ones.

- **[schema/case-sensitive-strings]** On MySQL `utf8mb4_unicode_ci` (the project default), string UNIQUE indexes are **case-insensitive**: `/Pets` and `/pets` are treated as duplicates. OpenAPI path templates and tag names are case-sensitive identifiers. **Why:** correct grants for case-differing paths/tags would silently collide in production. **How to apply:** add `->collation('utf8mb4_bin')` to any column holding case-sensitive string identifiers (`tag`, `path`). Guard with `if (in_array(DB::getDriverName(), ['mysql', 'mariadb'], true))` ‚Äî **include `mariadb`** (the framework reports it distinctly from `mysql`, and the project config defines both), so MariaDB also gets the binary collation; SQLite (binary UNIQUE by default) skips it to avoid an unknown-collation error.

- **[schema/audit-log-timestamps]** Use `->timestamp('created_at')->useCurrent()` (NOT NULL + DB default) for immutable audit tables, not `->nullable()`. **Why:** nullable `created_at` with no DB default lets any raw insert (bypassing Eloquent timestamping) create a row with no timestamp, weakening forensic ordering. **How to apply:** `useCurrent()` is SQLite-compatible and is the idiomatic Laravel pattern for append-only tables.

- **[schema/email-column-width]** When snapshotting a user's email into another table (e.g., auth_logs), use the **same width** as `users.email` (255), not 191. **Why:** 191 was the MySQL utf8mb4 index-length limit for the old MyISAM 767-byte row limit, but InnoDB 5.7+ / Laravel default uses 1000-byte, so 255 is safe. A 191-char snapshot can silently truncate or fail on a valid long email. **How to apply:** `$table->string('email', 255)` everywhere an email is stored, consistently with the `users` table.

- **[enum/http-verbs]** OpenAPI 3.x path-item objects support **8** HTTP methods: `get, put, post, delete, options, head, patch, trace`. Don't omit `TRACE`. **Why:** a TRACE operation in an upstream spec would be silently unrepresentable/ungrantable with a 7-verb enum. **How to apply:** `HttpVerb` must include `case Trace = 'TRACE'`; test `HttpVerb::values()` with `assertCount(8, ...)` and include `TRACE`.

- **[security/component-alias-same-type]** A reusable component that is a top-level `$ref` alias (Link, Example, Callback, and `components.links` entries) MUST point to a component of the **same type**. A `components.links` entry like `{$ref: '#/components/schemas/Internal'}` is malformed ‚Äî but if the pruner treats any non-link local `$ref` as "external, keep conservatively", it both (a) keeps the bogus link and (b) drains the targeted schema, leaking an ungranted component into a filtered spec. **Why:** the conservative "external refs survive" rule is correct only for *truly external* refs (other documents); a same-document `$ref` to the wrong component type is a leak vector, not an external dependency. **How to apply:** the rule must hold at **every** reachability site, not one ‚Äî here three: the inline-link survival check (`linkTargetSurvives`), the `components.links` **alias fixpoint**, and `drainReachability`'s example/link/callback alias branch. In each, a same-document fragment that doesn't start with the expected `<type>/` prefix ‚Üí drop (never mark surviving, never follow); only a `localFragment()===null` (external/other-doc) ref is kept conservatively. Tests: a link `$ref` to a schema is dropped without leaking the schema. Also add `scopes` to the keyword-name-map guard so an OAuth2 scope literally named `$ref` is walked as scope DATA, not chased. (Codex P1 on PR #16.)

- **[security/operation-bearing-ref-is-type-specific]** Operation-bearing components (`pathItems`, `callbacks`) are reachable only from structural positions ‚Äî but each position admits exactly **one** type: a Path Item `$ref` resolves only to another Path Item, a Callback `$ref` only to a Callback. A single boolean "allow operation-bearing here" is too coarse: it lets a callback path item (or a reusable `components.pathItems` entry) with a top-level `$ref` to `#/components/callbacks/Hidden` mark that callback reachable, keeping its ungranted operation (and any schema it references) in the non-admin spec. **Why:** cross-type structural refs (path-item‚Üícallback or callback‚Üípath-item) are malformed, not legitimate reuse; treating them as reachable is a leak. **How to apply:** make the allowance carry the *single legal type* (`pathItems` | `callbacks` | none), not a bool. Path-item context (the `$ref` branch + callback-path-item walk) passes `pathItems`; a Callback `$ref` passes `callbacks`; everything else passes none. In the guard, an owning component of an operation-bearing type that isn't the permitted one is dropped. (Codex P2 on PR #16, `eeea26e`.)

- **[uri/preserve-empty-path-segments]** When normalizing a URI path to compare two document URIs, resolve `.`/`..` per RFC 3986 ¬ß5.2.4 but **do not** collapse empty path segments. A naive `foreach (explode('/')) if ($seg==='') continue;` turns `//openapi.json` and `/openapi.json/` into `/openapi.json` ‚Äî so an **external** ref `https://host//openapi.json#/components/schemas/Internal` compares equal to the configured upstream `https://host/openapi.json` document, gets treated as same-document, and keeps the local `Internal` component in a non-admin spec even though nothing local referenced it. **Why:** distinct URI paths must stay distinct or `refBaseIsCurrentDocument()` mis-classifies a foreign document as local ‚Üí leak. **How to apply:** drop only the leading empty segment (from the leading `/`), keep all other empties, drop `.`, and let `..` pop the previous segment (name or empty); re-prepend `/` for absolute paths. Test with `//file` and `/file/` refs. **Refinement (RFC 3986 ¬ß5.2.4):** a *terminal* `.`/`..` segment leaves a **trailing slash** ‚Äî `/openapi.json/.` ‚Üí `/openapi.json/`, `/a/b/..` ‚Üí `/a/` ‚Äî so a naive "drop `.`" still collapses `.../openapi.json/.` onto the upstream and leaks. Append an empty output segment when the dropped/ascended segment is the last one. Test with `.../openapi.json/.` refs too. (Codex P2 chain on PR #16, `b610331`‚Üí`138d3b5`.)

- **[openapi/numeric-keyed-maps-decode-as-list]** A JSON object with digit-only, sequential keys (`{"0":‚Ä¶,"1":‚Ä¶}`) is indistinguishable from a JSON array after `json_decode($x, true)` ‚Äî PHP makes both a list, so `array_is_list()` returns true for a legitimately-keyed OpenAPI map. This bit the `examples` walk: an Example **Objects map** named `"0"/"1"` was misread as a JSON-Schema example **values array** and skipped, dropping its real `#/components/examples/‚Ä¶` refs ‚Üí dangling ref after pruning. **Why:** OpenAPI keyed maps (`examples`, and potentially others) can have numeric names; gating ref-collection on `array_is_list` silently drops them. **How to apply:** don't branch on `array_is_list` to decide map-vs-array for ref collection ‚Äî iterate entries either way and collect each entry's top-level `$ref` (a genuine values array's entries don't have a top-level `$ref` key, and we never recurse into an example value, so DATA stays skipped). Over-retaining a coincidental data-shaped `$ref` is the safe direction (no leak, no dangle). (Codex P2 on PR #16, `b610331`.)

- **[openapi/examples-meaning-is-contextual]** The `examples` key means two different things depending on the **object type** it sits in, and you cannot tell them apart by shape: inside a **Schema Object** it is the JSON Schema keyword (an ARRAY of raw example DATA); inside a **Media Type / Parameter / Header Object** it is a MAP of Example Objects (each possibly `{$ref: #/components/examples/X}`). A raw JSON Schema example datum can itself be `{"$ref":"#/components/schemas/Internal"}` ‚Äî that is DATA, not a reference ‚Äî so scanning schema `examples` for refs **leaks** an otherwise-unreferenced schema. Conversely, skipping list-shaped `examples` drops real refs from a numeric-keyed Example map. **Why:** the first naive fix (`! array_is_list`) caused the dangling-ref bug; over-correcting (always iterate) caused the leak bug ‚Äî shape alone is insufficient. **How to apply:** disambiguate by **context**, not shape. Thread an `$inSchema` flag through the reachability walk: set true when descending through the `schema` keyword or any JSON Schema applicator (`properties`/`$defs`/`definitions`/`patternProperties`/`dependentSchemas`/`items`/`prefixItems`/`additionalProperties`/`allOf`/`anyOf`/`oneOf`/`not`/`if`/`then`/`else`/‚Ä¶ ‚Äî monotonic downward, since a subschema is always a schema). Then `examples` is skipped when `$inSchema`, and treated as an Example map (iterating regardless of list-ness, for numeric keys) otherwise. Drain component `schemas` with `$inSchema=true`. This generalizes [[openapi-numeric-keyed-maps-decode-as-list]]. (Codex P2 chain on PR #16, `b610331`‚Üí`f5079e1`.)

- **[uri/percent-decode-fragments]** A URI fragment is percent-encoded (RFC 3986), so a JSON Pointer `$ref`/`operationRef` can legitimately arrive as `#%2Fpaths%2F~1admin%2Fget`. If you test the raw fragment against `/paths/‚Ä¶`/`/components/‚Ä¶` prefixes it won't match, so a same-document pointer to a **filtered-out** operation/component is mis-classified as *external* and kept ‚Äî leaking the link and the target's path name. **Why:** "external refs are kept conservatively" is right for other documents, but a percent-encoded local pointer is NOT external. **How to apply:** `rawurldecode()` the fragment in the one place that extracts it (`localFragment()`), so every caller sees `/paths/~1admin/get`. Use `rawurldecode` (not `urldecode` ‚Äî `+` must stay `+`); JSON Pointer `~0`/`~1` escapes are not percent-encoding and survive decoding for the existing `str_replace(['~1','~0'],‚Ä¶)` step. Test both directions (encoded ref to a filtered op dropped; to a granted op kept). (Codex P2 on PR #16, `f49068e`.)

- **[review/graphql-thread-pagination]** GitHub's `reviewThreads` connection caps `first`/`last` at **100**; once a long-lived PR accumulates >100 review threads, `reviewThreads(first:110)` errors `EXCESSIVE_PAGINATION` and a naive count silently breaks. **Why:** our monitor counted unresolved threads with a growing `first:N`. **How to apply:** to find *newly created* unresolved threads, query `reviewThreads(last:100)` (newest are at the end of the connection); to enumerate ALL, paginate with `first:100, after:$cursor` following `pageInfo.hasNextPage`/`endCursor`.

- **[security/keyword-meaning-is-positional]** Several reachability-bearing keys (`examples`, `security`, and the `$ref`/operation-bearing handling) mean different things by POSITION, and a Schema Object can carry an arbitrarily-named member that collides with an OpenAPI keyword. After threading `$inSchema`, audit **every** keyword branch in the walk for "is this still a real keyword inside a schema?": `examples` inside a schema is data (done); `security` is a Security Requirement ONLY as a direct child of an Operation Object (or the root, seeded separately) ‚Äî NOT in a schema, and crucially **not in a response/header/parameter/link either**. `! $inSchema` was *insufficient* for `security` (responses/headers are also `$inSchema=false`); it needed a precise `$securityIsRequirement` flag set true only when descending into an HTTP-verb child, reset for that operation's sub-objects. **Why:** a context flag fixes a CLASS of "member shaped like an OpenAPI keyword ‚Üí leak" bugs, but the *granularity* must match each keyword's real valid positions ‚Äî `examples` is schema-vs-not, `security` is operation-only. **How to apply:** when you add a context flag for one keyword, sweep the sibling branches AND pick each one's correct position predicate (don't assume the same flag fits). Generalizes [[openapi-examples-meaning-is-contextual]]. (Codex P2 chain on PR #16, `f5edc94`‚Üí`46acdab`.)

- **[openapi/empty-maps-must-be-json-objects]** OpenAPI map fields (`paths`, `components`, `responses`, ‚Ä¶) MUST serialize as JSON objects. In PHP an emptied associative array is `[]`, which `json_encode`s to `"[]"` ‚Äî an invalid document a strict validator/Scalar rejects. After filtering can empty a map (e.g. a no-grant or webhook-only user gets empty `paths`), force the object form: `$spec['paths'] = (object) []`. **Why:** non-empty assoc arrays already encode as objects, so only the *emptied* case is wrong ‚Äî and it only surfaces for the least-privileged users, so it's easy to miss in tests that always grant something. **How to apply:** normalize empty required-object maps to `(object) []` at the producer's boundary, AFTER any array-based internal processing (don't break code that reads them as arrays). In Pest, assert the object form with `->toEqual((object) [])` (an array `[]` is `!=` an empty stdClass, so this catches a regression). (Codex P2 on PR #16, `46acdab`.)

- **[security/parallel-walks-must-share-context]** When a guard depends on a context flag (here `$inSchema`), EVERY independent walk of the same data must compute that flag the same way ‚Äî or the walks disagree and the gap is a leak. The reachability core walks components twice: once for `$ref`s (drain) and once for `$anchor` declarations (anchor-owner map). After adding the schema-position guard to ref collection, the anchor scan still ran with `$inSchema=false`, so a `$anchor` buried in a non-standard `callbacks` annotation of an unreachable schema was attributed to it; a surviving `#anchor` ref then kept the hidden schema. **Why:** an anchor-owner map that includes anchors the drain would skip creates a reachability path the drain never intended. **How to apply:** thread the SAME context into every parallel walk (`collectReachableRefs` AND `collectAnchorNames` both take `$inSchema`; the component loop passes `$type==='schemas'` to both). After threading any new context flag, grep every walk/collect call site and verify each passes it consistently ‚Äî don't fix only the site the reviewer happened to point at. (Codex P2 on PR #16, `f7641d8`.)

- **[security/reusable-components-bypass-top-level-filtering]** (P1) Filtering applied at the TOP level (paths/webhooks operations dropped by tag/endpoint) does NOT cover the same operations when they live in a **reusable component** (`components.pathItems`) reached another way ‚Äî e.g. via a granted operation's callback or a link `operationRef`. The reachability prune keeps such a component WHOLE, so an extra operation in it (tagged for another audience) ships to the browser with its schemas. **Why:** top-level path items are filtered+inlined by `filterPathItemMap`/`resolvePathItem`, but the component SOURCE is only ever pruned for *reachability*, never for *grant* ‚Äî reachable ‚âÝ authorized. **How to apply:** filter operation-bearing component entries by grant too, not just reachability. Reusable path items have no canonical endpoint path, so only a TAG grant can authorize their operations ‚Äî drop the rest, then re-prune orphaned schemas. Keep the component's `$ref`-resolution source (used for top-level inlining) separate/untouched so endpoint-granted `$ref`'d paths still resolve fully. General rule: any place a component can carry an OPERATION (pathItems, callbacks) must apply the authorization filter, not assume the top-level pass covered it. (Codex P1 on PR #16, `33a1df5`.)
